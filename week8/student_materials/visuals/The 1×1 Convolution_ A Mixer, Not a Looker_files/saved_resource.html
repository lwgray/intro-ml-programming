The 1×1 Convolution: A Mixer, Not a Looker

This page starts with the fact most courses skip: every conv filter is a 3D brick, not a flat square. Once that clicks, "1×1 conv" stops being mysterious — it's the brick shrunk to a pin — and ResNet's two plumbing fixes follow in one step each.

1 · The missing fact

Every filter is a brick — it spans ALL input channels

When we say "a 3×3 filter" on a 6-channel input, the filter is actually 3×3×6 — a brick of 54 weights that reads a 3×3 neighborhood on every one of the 6 input maps at once. One brick always produces exactly one output map. Want 32 output channels? Use 32 bricks.

Check it against your own Week 8 model: layer 2 is "3×3" on 32 input channels, so each filter is really 3×3×32 = 288 weights (+1 bias) = 289, and 289 × 32 filters = 9,248 — exactly what model.summary() prints. The formula params = k_h·k_w · C_in · C_out is just "brick volume × number of bricks."

2 · So what's a 1×1 filter?

The brick shrunk to a pin

Set k = 1 and the brick becomes a 1×1×6 pin: it sees no neighbors at all — just the 6 values stacked at one pixel position, one from each channel's map (push a pin through the deck of 6 transparencies; it crosses 6 numbers). 3 · The channel change

256 → 64 isn't one filter — it's 64 pins, each making one map

On a 256-channel input, each 1×1 filter is a 1×1×256 pin: 256 weights, one weighted sum per pixel, one output map. Stack 64 such filters — each with its own 256 weights, each producing its own map — and the output has 64 channels:

Total: 256 × 64 = 16,384 weights (+64 biases) — the "16K" in the bottleneck below. In linear-algebra terms: one 64×256 matrix multiplying the 256-vector that lives at every pixel — the same matrix everywhere.
Why this is meaningful, not just cheap: the 256 input channels are 256 filters' opinions about each spot ("rim-ness: 3, fur-ness: 0, stripe-ness: 7…"). Each 1×1 filter is a learned recipe for a composite opinion — "wheel-ness = 0.4·rim + 0.2·spoke − 0.1·fur…". Sixty-four recipes turn a 256-word description into a 64-word one, at every pixel, without ever looking sideways.
4 · Problem 1

The skip can't add mismatched shapes

At every ResNet stage boundary the main path halves the map and doubles the channels (56²×64 → 28²×128). But the ⊕ needs to compute x + F(x) — and x is still the old shape. You can't add tensors that don't line up.

The 1×1 here is a shape adapter: stride 2 halves the map to 28², and 128 pins — each blending the old 64 channels its own way — produce the new channel count. The skip stays as close to "a bare wire" as the shapes allow.

5 · Problem 2

3×3 bricks at 256 channels are brutally expensive

Deep in the network, channels are wide — and brick volume is 9 × C_in per filter: one 3×3 conv at 256→256 costs 3·3·256·256 ≈ 590K weights. The bottleneck block uses pins to compress, does the spatial work with small bricks in the cheap space, then re-expands:

The ELI5: 256 channels = a committee of 256 opinions at every pixel. The first set of pins elects 64 delegates (each a learned blend of all 256 opinions), the expensive 3×3 meeting happens with just the delegates, and the last pins report back to all 256 seats. Only the middle step ever looks at neighboring pixels — and it runs in the cheap room. This sandwich is why ResNet-50/101/152 exist; ResNet-18/34 use plain 3×3 blocks and simply couldn't afford to go deeper.

A filter is a brick that spans all input channels and always yields one output map — a 1×1 filter is that brick shrunk to a pin: C_in weights blending the stacked scores at each pixel. 64 pins = 64 new channels. ResNet uses pins twice: as a shape adapter so the skip's add works, and as the compress-work-expand sandwich that made 152 layers affordable.
